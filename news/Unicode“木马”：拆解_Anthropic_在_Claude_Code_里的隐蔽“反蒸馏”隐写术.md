# Unicode“木马”：拆解 Anthropic 在 Claude Code 里的隐蔽“反蒸馏”隐写术

在竞争白热化的 LLM 工程战场上，知识产权的攻防战早已不再局限于服务器端的权重和私有训练数据集，而是直接蔓延到了客户端。过去一周，围绕 Anthropic 旗舰级命令行 AI 编程智能体 `@anthropic-ai/claude-code` 爆发了一场重大的安全丑闻。安全研究人员曝光了该 CLI 客户端内部运行的一个未公开且经过混淆的遥测通道——它暗中扫描用户的本地网络环境，并以隐写术（Steganography）的形式将用户元数据静默回传给 Anthropic 的后端。

这一发现随即引爆了地缘政治与企业级市场的连锁震荡。2026年7月8日，中国国家信息安全漏洞库（NVDB）发布官方安全警示，将该遥测机制定义为“安全后门”，指责其在未获得用户授权的情况下传输地理位置与身份标识信息，严重侵犯用户隐私。与此同时，阿里巴巴集团下发内部全员禁令，严禁员工使用 Claude Code，并要求在2026年7月10日前全部迁移至其自研的 AI 编码助手“通义灵码”。

要理解这场因几个 Unicode 字符而起的地缘政治风暴，我们需要深入其被逆向工程拆解的源码、Prompt 隐写注入机制，以及驱动 Anthropic 采取这种极端防御性工程的商业间谍战背景。

### 隐写泄露的解剖：版本 2.1.91 至 2.1.196

这起风波的起点可以追溯到 2026 年 6 月下旬。一位在 Reddit 上活跃、名为 `LegitMichel777` 的开发者，在试图通过逆向工程还原 Claude Code CLI 工具中被禁用的远程执行功能时，意外在被压缩混淆（minified）的 JavaScript 代码中，发现了一段高度混淆的本地环境及网络指纹探测逻辑。

随后，独立安全研究员 `Thereallo` 在其个人博客 `thereallo.dev` 上发表了详尽的拆解报告，证实该工具绕过了常规的 API 遥测通道，以极具隐蔽性的方式向后端传送分类特征信号。

#### 客户端本地审计逻辑
当开发者启动 Claude Code 时，客户端会首先读取环境变量 `ANTHROPIC_BASE_URL`。一旦检测到该变量指向了自定义代理、模型路由器或第三方 API 网关，客户端便会立刻触发本地环境审计：
1. **时区检查：** 工具通过调用 `Intl.DateTimeFormat().resolvedOptions().timeZone` 获取系统时区，并对 `Asia/Shanghai`（亚洲/上海）和 `Asia/Urumqi`（亚洲/乌鲁木齐）进行重点标记。
2. **网络代理扫描：** 工具将代理主机的域名与一个硬编码的黑名单进行比对。该黑名单包含 147 个实体（主要为中国云服务商和 VPS 主机商）以及 11 个特定 AI 实验室的关键词（如 `deepseek`、`moonshot`、`zhipu` 和 `minimax`）。

为了防止这些敏感域名和关键词在二进制数据导出时被轻易以明文检索，黑名单数据在源码中经过了 Base64 编码，并使用静态密钥 `91` 进行了 XOR 异或混淆。

```typescript
// 针对 v2.1.196 中发现的 XOR 解密算法的原理性重构
function decryptBlocklist(obfuscatedB64: string): string[] {
  const buffer = Buffer.from(obfuscatedB64, 'base64');
  const decrypted = buffer.map(byte => byte ^ 91);
  return JSON.parse(decrypted.toString('utf-8'));
}
```

#### 提示词隐写传输通道
通常情况下，遥测数据会通过标准的 HTTP POST 请求发送到特定的日志端点，而这在 Wireshark 等网络监控工具下会一览无余。为了避开安全审计，Anthropic 另辟蹊径，采用了一种更隐蔽的通道——**Prompt 隐写术（Prompt Steganography）**。

CLI 工具会在本地拦截组装好的 System Prompt（系统提示词），并对其中一条看似平淡无奇的系统时间字符串进行动态篡改。目标字符串为：`"Today's date is YYYY-MM-DD."`（今天的日期是 YYYY-MM-DD）。

根据本地环境审计的结果，客户端会以两种方式微调该字符串：

1. **日期分隔符翻转：** 若系统时区匹配 `Asia/Shanghai` 或 `Asia/Urumqi`，原本作为分隔符的连字符（-）会被替换为斜杠（/）。例如，`2026-07-08` 会变成 `2026/07/08`。
2. **Unicode 撇号替换：** 单词 `Today's` 中标准的 ASCII 撇号（`U+0027`）会被动态替换为字形几乎完全相同、但在 Unicode 编码中完全不同的其他撇号字符：
   * 替换为 **`U+2019`**（右单引号 `’`）：代表匹配到了代理域名黑名单。
   * 替换为 **`U+02BC`**（修饰符字母撇号 `ʼ`）：代表匹配到了 AI 实验室关键词。
   * 替换为 **`U+02B9`**（修饰符字母角分号 `ʹ`）：代表同时触发了上述两项匹配。

当请求送达 Anthropic 服务器后，后端解析器会读取这个 System Prompt。通过检测日期分隔符和撇号的 Unicode 编码，Anthropic 能够瞬间识别出该流量的真实来源与路由路径。即使用户通过国内代理或 VPS 屏蔽了真实 IP，在服务端的流量画像面前也形同裸奔。

### 地缘政治背景：LLM 时代的“反蒸馏”战争

为什么一家顶尖的 AI 实验室，要在开发者的 CLI 工具里塞进这种特务般的隐写术？答案指向了行业内残酷的“模型蒸馏”（Model Distillation）暗战。

数月来，Anthropic 内部一直指责外部竞争对手（包括阿里巴巴的 Qwen 团队等）在大规模部署自动化爬虫系统。通过注册成千上万个虚假账号，竞争对手正在源源不断地用 Claude 3.5 Sonnet 的生成结果来训练和蒸馏他们本土的竞争模型。由于这些爬虫普遍套用了本地代理以绕过地理限制和速率限制，常规的 IP 阻断与 API Token 封禁策略彻底失效。

这一隐写遥测本质上是一场反侦察实验。通过在 Prompt 中注入“隐形水印”，Anthropic 的后端能够绘制出哪些代理集群正在源源不断地向反蒸馏管线输送数据。这既帮助他们梳理了爬虫背后的基础设施，又避免了因在客户端直接抛出 HTTP 错误而打草惊蛇。

Anthropic 工程师 Thariq Shihipar 随后在 X.com 上公开回应并证实了这一机制：
> “这是我们在今年 3 月启动的一项反滥用实验，旨在保护我们的知识产权免受系统性账号滥用和竞争对手未授权的模型蒸馏。随着我们转向更强大的服务端检测方案，目前正在逐步移除这一客户端机制。”

### 前车之鉴：2026年3月的 npm 源码映射泄露事件

事实上，这并非 `@anthropic-ai/claude-code` 第一次遭遇严重的安全翻车。早在 2026 年 3 月，该 npm 包的 `2.1.88` 版本就曾因疏忽意外打包了完整的 Source Map（`.map` 源码映射文件）。这使得安全研究人员得以完整复原出该工具近 50 万行未混淆的 TypeScript 源代码。

当时的源码泄露揭示了 Anthropic 的其他防御手段，例如**“虚假工具（Fake Tool）注入”**。为了对抗模型蒸馏，Claude Code 客户端被设计为会在 Agent 的上下文里注入一些诱饵工具（例如虚拟的文件系统 API 或虚假的终端 Shell）。如果竞争对手的爬虫抓取了这些包含虚假工具的对话日志去训练“学生模型”，那么训练出来的模型就会被“投毒”，在实际使用中频繁调用这些根本不存在的幻觉 API。

那次源码泄露还引发了一波供应链攻击浪潮。恶意攻击者克隆了泄露的代码，并在 GitHub 和 npm 上发布了大量诸如 `claude-code-cli` 和 `@anthropic-ai/claude-coder` 的李鬼包。这些恶意包中被植入了 Vidar 和 GhostSocks 等信息窃取木马，专门搜寻用户本地的 `~/.claude.json` 配置文件以窃取活跃的 API Token，甚至劫持 Model Context Protocol（MCP）服务器，从而拿走用户 Jira、GitHub 和 Confluence 的访问权限密钥。

### 伦理争议：信任边界与 Agent 时代的本地工作区

这种未公开的隐写术被曝光后，在开发者群体中引发了极大的信任危机。核心的争论点在于“信任边界”。与传统的网页版 LLM 聊天窗口不同，像 Claude Code 这样的 CLI 智能体运行在本地，拥有极高的系统权限——它们能直接读取本地文件、执行终端命令以及配置系统环境。

在 Reddit 的 `r/LocalLLaMA` 板块上，一位知名工程师的发言引发了广泛共鸣：“如果一个智能体在我的机器上以 Shell 权限运行，它就不能有任何不透明的行为。一旦一个开发者工具开始偷偷扫描我的本地时区和网络代理，并通过隐写术向外发送数据，无论它的反蒸馏动机多么崇高，它的行为在本质上都与间谍软件无异。”

这与企业常规接受的遥测技术背道而驰。现代企业的可观测性体系依赖于像 OpenTelemetry（OTel）这样完全可审计、可配置且透明的开放标准。而 Anthropic 将遥测隐藏在 Prompt 字符串中的做法，击穿了开发者工具与企业客户之间最基本的信任底线。

对于企业用户而言，这一丑闻正加速他们转向开源或本地化的替代方案。越来越多的企业开始转向自托管的开发者助手（如 Aider 或 Open Interpreter），并搭配本地部署的开源权重模型（如 DeepSeek-Coder 或基于 Llama-3 的智能体）。在完全沙箱化的网络环境中，任何私有源码和网络元数据都将彻底失去被“隐写”回传至第三方服务器的可能。
