# 走出铁笼，奔向云端：拆解英伟达 Halos 与人形机器人分布式安全“豪赌”

几十年来，工业自动化的金科玉律极其简单粗暴：将人类与重型机械进行物理隔离。在汽车装配线和物流枢纽中，机器人被限制在坚固的钢制安全笼内运行。一旦有人类闯入，物理联锁装置会立即触发并瞬间切断电源。然而，随着以 Agility Robotics 的 [Digit](https://agilityrobotics.com/) 为代表的通用人形机器人的兴起，这一传统范式正在被彻底粉碎。为了实现在无结构化的复杂环境中与人类并肩工作的承诺，人形机器人必须走出安全笼。

为了填补这一安全红沟，英伟达在 Automate 大会上推出了“**Halos for Robotics**”（机器人安全光环）平台。作为一个全栈式功能安全平台，Halos 融合了工业级硬件、实时操作系统（RTOS）以及一种被称为“**Outside-In Safety Blueprint**”（由外至内的安全蓝图）的分布式感知范式。

然而，将事关生命安全的核心控制逻辑从机器人本体剥离、并转移到厂区网络中，在机器人工程师、传统控制系统专家和 AI 学者之间引爆了一场激烈辩论。分布式安全系统真的能做到绝对的“确定性”（Deterministic）吗？英伟达此举是否在为具身智能（Physical AI）引入致命的单点网络故障隐患？

### 架构的分野：确定性 QNX 与概率性 AI 的博弈

英伟达 Halos 平台的核心是 [NVIDIA IGX Thor](https://www.nvidia.com/en-us/edge-computing/products/igx/) 开发套件。这是一台通过了功能安全认证的工业级计算机，旨在实现高负载 AI 任务与硬实时（Hard Real-Time）控制回路的同芯片并行运行。

```
+-----------------------------------------------------------------------------------+
|                                 NVIDIA IGX THOR                                   |
|                                                                                   |
|  +-----------------------------------+     +-----------------------------------+  |
|  |             AI 应用分区            |     |            安全关键分区           |  |
|  |               (Linux)             |     |        (QNX OS for Safety)        |  |
|  |  - 深度学习模型                   |     |  - Halos 核心                     |  |
|  |  - 概率性目标检测                 |     |  - 确定性安全逻辑                 |  |
|  +-----------------+-----------------+     +-----------------+-----------------+  |
|                    |                                         |                    |
|                    +--------------------+--------------------+                    |
|                                         |                                         |
|                                     Hypervisor                                    |
|                                         |                                         |
|  +--------------------------------------v--------------------------------------+  |
|  |                            芯片级功能安全岛 (FSI)                             |  |
|  |             - 双核锁步 (DCLS) Arm Cortex-R52 内核                              |  |
|  |             - 独立电压/电源轨与错误管理                                        |  |
|  +-----------------------------------------------------------------------------+  |
+-----------------------------------------------------------------------------------+
```

在过去，根据 **IEC 61508**（最高 SIL 3 等级）和 **ISO 13849**（性能等级 PL e, 第 4 类）等功能安全标准，将本质上属于概率性且存在“幻觉”风险的深度学习模型，与要求 100% 确定性的安全控制逻辑运行在同一颗处理器上，在工程上几乎是不可能实现的。而 Halos OS 通过基于 Hypervisor 的虚拟化技术解决了这一冲突，将非安全计算域（Linux）与运行 **QNX OS for Safety 8.0** 或 **FreeRTOS** 的安全关键域进行了严格隔离。

至关重要的是，IGX Thor 芯片内置了“**功能安全岛**”（Functional Safety Island, FSI）。FSI 是一个硬件隔离的子系统，由配置为**双核锁步**（DCLS）的 **Arm Cortex-R52** 内核集群驱动。FSI 运行在完全独立的电压和电源轨上，负责监控主 CPU 和 GPU 复合体的运行状况、检测硬件故障，并确保“**免受干扰**”（Freedom From Interference, FFI）。即便庞大的神经网络导致 Linux 分区崩溃，QNX 安全分区和 FSI 仍能保持正常运转，从而确保机器人能够安全、受控地执行紧急停机。

### “由外至内”安全蓝图：功耗与延迟的权衡

Halos 架构中最具争议的元素莫过于**“由外至内”安全蓝图**（Outside-In Safety Blueprint）。传统的“由内而外”安全机制完全依赖机器人自身的车载传感器套件——如深度相机、激光雷达（LiDAR）和惯性测量单元（IMU）。然而，在繁忙的仓库环境中，人形机器人车载传感器会面临严重的遮挡问题：一叠托盘或另一台机器人就能轻而易举地挡住它的视线，使其无法察觉正在靠近的人类。

英伟达的“由外至内”方案将环境监控任务卸载到了安装在厂区天花板上的监控摄像头上。这些外部传感器将空间数据实时输入到中央边缘计算节点，该节点运行着一个**由外至内安全代理**（Outside-In Safety Agent），构建出全局的 3D 占据网格图，并在机器人周围投影出“动态虚拟安全围栏”。

```
                                 [天花板摄像头]
                                       |
                                       | (基于 RDMA 的极低延迟数据流)
                                       v
                                 [中央安全节点]
                                       |
                                       | (无线遥测回路)
                                       v
+--------------------------------------v--------------------------------------+
|                                 车载控制回路                                 |
|                                                                             |
|  +--------------------------+                 +--------------------------+  |
|  |         安全核心         |                 |        执行器控制        |  |
|  |   (监控心跳与无线遥测)    |================>|   (紧急减速 / 切断电源)   |  |
|  +------------^-------------+    (硬件直连)   +--------------------------+  |
|               |                                                             |
|               +----------------------+                                      |
|                                      |                                      |
|                       +--------------+---------------+                      |
|                       |         车载传感器套件       |                      |
|                       |  (激光雷达、深度相机、IMU)   |                      |
|                       +------------------------------+                      |
+-----------------------------------------------------------------------------+
```

支持者认为，这是人形机器人走向落地的唯一可行路径。“对于依靠电池供电的人形机器人，其车载算力预算极其紧张，通常被限制在 150-200W，”Reddit 上一位知名的机器人硬件工程师指出，“如果让机器人在本地运行高保真 3D 空间建模和实时避障算法，电池在一个小时内就会被消耗殆尽。将感知计算外包，纯粹是受物理规律所迫。”

然而，传统的安全工程师对此持高度怀疑态度。根据 **ISO 10218**（工业机器人安全标准），安全相关控制回路必须具备确定性。怀疑者指出，依赖 Wi-Fi 或 5G 网络会带来丢包和延迟抖动问题。一旦厂区摄像头失准，或者无线数据包丢失，机器人的反应时间就会受到严重影响。

为了克服这些网络瓶颈，英伟达采用了硬件加速的通信路径。该平台利用 **NVIDIA ConnectX RDMA**（远程直接内存访问）和 **RTX GPU Direct** 技术绕过操作系统的 TCP/IP 协议栈，以亚毫秒级的延迟将传感器数据直接推送到 GPU 显存中。

当发生丢包时，Halos 安全核心（Halos Safety Core）的设计机制是直接丢弃延迟的遥测数据。系统不会基于过期的数据做出决策，而是依赖一个由专用 **SEI 守护进程**（daemon）监控的持续心跳信号。如果心跳丢失或延迟超过了硬性阈值（通常为 50-100毫秒），系统将立即触发故障保护（Fail-safe）状态：机器人会重新启用车载安全传感器，退回到局部的、保守的“速度与距离监控”模式，或者执行符合 Category 1 标准的受控停机。

### 硬件级安全兜底：与传统安全 PLC 的协同

无论 AI 模型多么先进，神经网络本质上依然是概率系统，无法单独通过功能安全认证。根据 **ISO 10218-2** 标准，人机协作作业必须通过经过安全评级的控制器来强制划定安全区域。

英伟达 Halos 的解决方案是与来自 Pilz（皮尔磁）或 Rockwell Automation（罗克韦尔自动化）等厂商的传统安全级**可编程逻辑控制器（PLC）**进行协同。

```
                                  [AI 模型]
                                (Linux 分区)
                                      |
                                      | (软件推理 / 分类)
                                      v
                             [NVIDIA Halos Core]
                              (QNX / FSI 安全域)
                                      |
                                      | (安全诊断与心跳)
                                      v
                             [载板安全 MCU]
                                      |
                                      | (硬件直连输出)
                                      v
                            [传统安全 PLC]
                    (运行 PROFIsafe / CIP Safety)
                                      |
                                      | (硬件直连急停 / 继电器)
                                      v
                           [执行器电源接触器]
```

IGX Thor 载板集成了一个专用的安全微控制器（MCU），它通过工业总线协议（如基于 Ethernet/IP 的 **PROFIsafe** 或 **CIP Safety**）直接与工业安全 PLC 交互。该安全 MCU 与安全 PLC 的输入通道进行硬件直连。

即使运行在 Linux 分区中的 AI 模型做出漏报预测（例如，将站在托盘后面的工人误分类为纸箱），系统依然能依靠这层硬件直连的兜底机制保障安全：
1. 安全 PLC 持续监控来自 IGX Thor 安全 MCU 的诊断心跳信号。
2. 一旦 AI 系统未能及时更新其安全令牌（Safety Token），或 FSI 检测到内部软件故障，安全 MCU 就会断开给 PLC 的硬件信号。
3. 安全 PLC 会瞬间通过安全接触器切断机器人主执行器线圈的电源（执行 IEC 60204-1 标准下的 Category 0 急停）。

这种架构确保了即使 AI 发生故障，硬件层面的边界依然不可逾越，从而维持了对 **ISO 13849-1** PL d/e 标准的合规性。

### 评测新前沿：ANAB 认可与 AI 系统检测实验室

具身智能商业化面临的最大绊脚石之一，就是缺乏标准化的验证框架。如果一个系统的输出是由神经网络而非静态代码生成的，你该如何对其进行安全认证？

为了解决这一难题，英伟达建立了 **Halos AI 系统检测实验室**（Halos AI Systems Inspection Lab）。该实验室近期成为全球首个获得美国国家标准协会合作机构——**ANSI 国家认可委员会（ANAB）**依据 **ISO/IEC 17020** 标准（《合格评定-各类执行检验机构的运作要求》）认可的 AI 系统检测项目。

与逐行审查代码的传统做法不同，该实验室将具身智能系统视作一个整体进行检测。其验证流程融合了功能安全（**IEC 61508**）、网络安全（**ISO/IEC 21434**）以及 AI 系统完整性（**ISO/IEC TR 5469**）。通过将集成后的软硬件栈置于标准测试向量下进行压力测试，实验室能帮助诸如 Agility Robotics 的合作伙伴获取预评估报告，从而大幅缩短 TÜV Rheinland 以及 UL Solutions 等国际权威第三方机构的最终认证周期。

### 一线实测：Agility Digit 上的安全协议

Agility Robotics 的 **Digit** 人形机器人是英伟达 Halos 平台的主要试验台。Digit 也有望成为首款将 Halos OS 直接集成到控制系统中的量产人形机器人。

针对 Digit 开发的安全测试协议分为两个截然不同的阶段：

1. **闭环仿真验证（Closed-Loop Simulation Validation）**：在软件部署到物理硬件之前，整个安全协议栈会在 **NVIDIA Isaac Sim** 中完成验证。仿真器会对传感器的物理特性、网络延迟以及仓库的物理动力学进行精确建模。Digit 的“感知到动作（perception-to-action）”安全闭环要经受数千次随机边缘场景的考验——包括突然的人类闯入、传感器镜头遮挡以及无线连接中断等，以验证 Halos 核心在任何情况下都能一致性地触发正确的故障保护状态。
2. **物理闯入测试（Physical Intrusion Testing）**：在物理测试环境中，Digit 必须接受严苛的动态安全审计。测试协议会精确测量整个控制链路的端到端延迟：从人类踏入天花板摄像头监控的活动区域那一刻起，经历 ConnectX RDMA 数据传输、QNX 安全控制器处理、安全 MCU 向 PLC 发送信号，直至 Digit 的执行器最终完成物理减速。

其核心目标是验证：在人类触及机器人动作包络线（Reach Envelope）之前，机器人能否可靠地实现完全静止，从而严格符合 **ISO/TS 15066** 标准中规定的“速度与距离监控”公式。

### 工业界的裁决

英伟达的 Halos 平台代表了人形机器人与具身智能商业化道路上的重大突破。通过将概率性的 AI 引擎与确定性的安全核心相分离，并引入 ANAB 认可的检测标准，英伟达正在为安全的人机协作奠定行业基石。

然而，对“由外至内”模式的依赖依然是一场经过精密计算的“豪赌”。虽然将高保真空间模型计算外包解决了人形机器人的功耗瓶颈，但这将前所未有的信任押注在了厂区的无线基础设施上。随着 Digit 逐步迈向生产环境，整个工业机器人行业都将密切注视：这种分布式安全模型是否真的能够应对工厂车间里那些混乱、充满未知与不确定性的复杂现实。
